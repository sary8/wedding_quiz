# 「curlだと200なのにブラウザだと画像が表示されない」の原因がCORPだった話

結婚式の余興用に、リアルタイムクイズアプリを個人開発しています。
無料枠にこだわった結果、フロントは Azure Static Web Apps、バックエンドは App Service という別オリジン構成になりました。
ドキュメントと手順記事を継ぎ接ぎしながらなんとかデプロイまで漕ぎ着けて、いざ動作確認！というタイミングで気づきました。
クイズの選択肢に付けた画像が、全部壊れたアイコンになっています。

APIは生きています。
クイズの作成もスマホからの参加もできて、壊れているのは画像だけ。
curlで画像URLを叩くと200が返ってきて、Content-Type も image/jpeg で、バイト数も合っています。
自動テストも全部通っています。
サーバー側の検証はすべて正常なのに、ブラウザだけが表示してくれない。
ここから原因に辿り着くまで、2時間かかりました…

## 先に結論

- 犯人はレスポンスヘッダの **Cross-Origin-Resource-Policy: same-origin**（CORSではなく**CORP**）
- 付けていたのは、Honoの `secureHeaders()` ミドルウェアのデフォルト値
- CORPを強制するのはブラウザだけなので、curlにも自動テストにも一切映らない
- メディア配信のレスポンスにだけ `Cross-Origin-Resource-Policy: cross-origin` を明示して解決

対処のコードだけ見たい方は、「解決：メディア配信だけを開ける」まで読み飛ばしてください。
ここから下は、そこに辿り着くまでに仮説を2回外した記録です。

## 1つ目の仮説：相対URLが悪い

バックエンドは画像URLを `/api/media/xxx.jpg` という相対パスで返していました。
同一オリジンで動かしている限り、これで何の問題もありません。
ローカル開発でも問題は出ません。Viteの開発サーバーが `/api` をバックエンドにプロキシしてくれるからです。

ところが本番は別オリジンです。
Static Web Apps 上のページが相対パスの画像を読みに行くと、行き先はバックエンドではなく Static Web Apps 自身になります。
そこに `/api/media/...` という静的ファイルはないので、404になります。

正直、この時点では「解決した！」と思っていました。
たしかにバグですし、症状も全部説明できるからです。
画像を表示している箇所を洗い出して、ビルド時に埋め込んだAPIオリジンを前置して絶対URLに解決する修正を入れました。
URLサニタイズ関数を通す方針にして、そこを通さず素通しでレンダリングしていた箇所も8ファイル見つかったので、まとめて直しました。

修正をデプロイして、管理画面を開きました。

画像は壊れたままでした。

## 2つ目の仮説：古いバンドルが動いているだけ

ここでは、まだ慌てていませんでした。
思い当たる節があったからです。
SPAは一度読み込むと、リロードするまで古いJavaScriptのまま動き続けます。
修正のデプロイをまたいで開きっぱなしだったタブなら、修正前のコードが画像を取りに行って失敗するのは当然です。
今度こそ説明がついた、と思いました。

スーパーリロードしても、直りませんでした。

仮説を2回続けて外して、思い当たる原因が尽きました。
ここで当てずっぽうをやめて、疑えるものを一個ずつ検証することにしました。

## 切り分け：どこまでが正しいのか

サーバー側から順に確かめていきました。

- DBから実在の画像ファイル名を取り出して、配信URLを直接叩く → 200、image/jpeg。**サーバーは正しい**
- デプロイ済みのJavaScriptチャンクをダウンロードして中身を開く → 絶対URL化のコードが、APIオリジンの文字列ごと焼き込まれている。**配信中のフロントも正しい**
- ページに付く Content-Security-Policy の `img-src` を確認する → バックエンドのオリジンは許可済み。**CSPも正しい**

サーバー側は3回調べ直して、どこにも異常がありませんでした。

消去法で残るのは、「curlにも自動テストにも現れず、ブラウザにだけ影響する何か」です。
まずCORSを疑ったのですが、調べてみると `<img>` タグによる画像の読み込みは no-cors のリクエストで、そもそもCORSの検査対象ではないようです。

※CORSとCORPは向きが逆です。CORSは「リクエストした側のオリジンに応答の中身を読ませてよいか」をサーバーが許可する仕組みで、CORPは「リソースを埋め込もうとするページのオリジン」をリソース側が制限する仕組み。`<img>` の表示はCORSの許可を必要としない代わりに、CORPの制限を受けます。

半信半疑のまま、画像URLのレスポンスヘッダを、今度は上から一行ずつ読みました。

```
HTTP/1.1 200 OK
Content-Type: image/jpeg
Cross-Origin-Resource-Policy: same-origin
```

見慣れないヘッダが一つ混ざっていました。
CORSじゃなくて、CORP。
恥ずかしながら、このとき初めて存在を知りました。

## CORPはブラウザだけが強制する

急いで調べた内容をまとめます。
**Cross-Origin-Resource-Policy（CORP）** は、リソースの側が「自分をどのオリジンのページに埋め込んでよいか」を宣言するレスポンスヘッダです。
`same-origin` が付いていると、別オリジンのページの `<img>` から読み込まれたとき、ブラウザは応答を受け取ったうえで、描画せずに捨てます。
Spectre系の攻撃（クロスオリジンのリソースを自プロセスに読み込ませて盗み見る手口）への防御として導入された仕組みで、強制するのはブラウザだけです。
curlはこのヘッダを素通ししますし、fetchのcorsモード通信も対象外なので、サーバーサイドの検証には一切映りません。
2時間悩んだ「curlでは200、ブラウザでは壊れる」という食い違いは、このヘッダの仕様どおりの動作でした。

では、誰がこのヘッダを付けたのか。
使っているWebフレームワーク Hono の `secureHeaders()` ミドルウェアが、デフォルトで `same-origin` を付けていました。
セキュリティ強化のつもりで、自分で入れたミドルウェアです。

**犯人、自分やんけ…**

フォローしておくと、入れた当時はフロントもAPIも同一オリジンで配信していたので、正しくて、かつ無害なデフォルト値でした。
別オリジン構成に移行したとき、このデフォルト値の前提が変わったことに気付いていなかった、というのが敗因です。

## 解決：メディア配信だけを開ける

画像や動画は、別オリジンのフロントに埋め込まれるのが正規の用途です。
なので、メディア配信のレスポンスにだけ `Cross-Origin-Resource-Policy: cross-origin` を付ければOKです。

ここでもう一つ罠がありました。
`secureHeaders()` はレスポンス確定後にヘッダを付けるので、ルート側で `cross-origin` を指定しても `same-origin` に上書きされてしまいます。
ミドルウェア側のCORP付与を止めて、メディア配信のルートで明示的に付ける形に落ち着きました。

```ts
// ミドルウェア: CORPの一括付与をやめる
app.use(
  secureHeaders({
    crossOriginResourcePolicy: false,
    // ...他の設定はそのまま
  })
);

// メディア配信レスポンス: 埋め込みを明示的に許可
return new Response(data, {
  headers: {
    "Content-Type": contentType,
    "Cache-Control": "public, max-age=31536000, immutable",
    "Cross-Origin-Resource-Policy": "cross-origin",
  },
});
```

JSONを返すAPIからはCORPが消えることになりますが、JSONは `<img>` にも `<script src>` にも意味のある形で埋め込めないので、実質的に失うものはありません。

検証のやり方も反省して、一つ変えました。
ルート単体のユニットテストだけでなく、ローカルで実サーバーを起動して、ミドルウェア通過後の最終ヘッダに `cross-origin` が残ることを確かめてからデプロイしました。
ミドルウェアの上書きは、ルート単体のテストには映らないんですよね。
今度こそ、ブラウザに画像が並びました。

## 「curlで通る」が保証してくれる範囲

curlの200は、「サーバーが正しく応答している」ことしか保証してくれません。
その応答をページに埋め込めるかどうかは、ブラウザが別のルールで決めています。
2時間かけて手に入れた整理はこうです。

- **CORS（Access-Control-Allow-Origin など）**：別オリジンからの「読み取り」をリクエスト側に許可する仕組み。fetch や XHR に効く
- **CORP（Cross-Origin-Resource-Policy）**：リソースの側が「埋め込まれてよい」オリジンを制限する仕組み。`<img>` や `<script>` に効く
- **CSP（Content-Security-Policy）**：ページの側が、読み込んでよいリソースの出所を制限する仕組み

同じ「画像だけ壊れる」症状は、CSPの `img-src` 不足でも起きます。
なので、仮説を並べる前に開発者ツールのコンソールとネットワークタブを開くのが、結局いちばん速いです。
サーバー側だけを2時間調べ続けた私が言うので、間違いないです…

## さいごに

今回いちばんの教訓は、「構成を変えたら、セキュリティ系ミドルウェアのデフォルト値を読み直す」です。
同一オリジンで無害だった設定が、別オリジン構成では障害の原因になりました。
同種のヘッダにはCOOPとCOEPというのもあるらしく、今回の構成ではまだ問題になっていませんが、別オリジン構成を続ける限り、いつか確認する日が来る気がしています。

同じ症状で検索して来た方の2時間が、この記事で10分になればうれしいです。

## おまけ：curl -I で二次遭難しかけた

白状すると、最初のヘッダ確認は `curl -I`（HEADリクエスト）でやって401が返ってきて、数分間「認証の問題か？」と原因を取り違えました。
このAPIは認証ミドルウェアがGETのメディア配信だけを公開ルートとして通す作りで、HEADは対象外だったのが理由です。
測定のつもりで打ったリクエストが、本番のリクエストと別物になっていることもある、という小さな学びでした。
