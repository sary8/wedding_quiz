# curlでは200が返るのにブラウザで画像が壊れるとき、Cross-Origin-Resource-Policy を疑う

結婚式の余興用に、リアルタイムクイズアプリを作っている。
フロントエンドは Azure Static Web Apps、バックエンドは App Service という、無料枠で組むとよくある別オリジン構成である。
デプロイを終えて管理画面を開くと、クイズの選択肢に付けた画像が、すべて壊れたアイコンになっていた。

APIは生きている。
クイズの作成もスマホからの参加もできて、壊れているのは画像だけだ。
curlで画像URLを叩くと200が返り、Content-Type は image/jpeg、バイト数も合っている。
自動テストもすべて通っている。
サーバーは「正しく返した」と言い、ブラウザは表示を拒む。
どちらかが嘘をついている。

## 最初の容疑者は相対URL

バックエンドは画像URLを `/api/media/xxx.jpg` という相対パスで返していた。
同一オリジンで動かしている限り、これで何の問題もない。
ローカル開発でも問題は出ない。
Viteの開発サーバーが `/api` をバックエンドにプロキシしてくれるからだ。

ところが本番は別オリジンである。
Static Web Apps 上のページが相対パスの画像を読みに行くと、行き先はバックエンドではなく Static Web Apps 自身になる。
そこに `/api/media/...` という静的ファイルはないから、404になる。

これは実際にバグだった。
画像を表示している箇所を洗い出し、ビルド時に埋め込んだAPIオリジンを前置して絶対URLに解決する修正を入れた。
URLサニタイズ関数を解決の一点に選び、そこを通さず素通しでレンダリングしていた箇所も8ファイル見つかって、まとめて直した。

これで直ったに違いない。
デプロイして、管理画面を開いた。
壊れたアイコンが、同じ顔で並んでいた。

## 「古いバンドルが動いている」という思い込み

次に疑ったのは、開きっぱなしのタブである。
SPAは一度読み込むと、リロードするまで古いJavaScriptのまま動き続ける。
修正のデプロイをまたいで開いていたタブなら、修正前のコードが画像を取りに行って失敗するのは当然の帰結だ。
今度こそ説明がついた、と思った。

スーパーリロードしても、直らなかった。

ここで、頭の中の仮説を並べるのをやめて、証拠を集める方に切り替えた。

## では、どこまでが正しいのか

サーバー側から順に確かめた。

- データベースから実在の画像ファイル名を取り出し、配信URLを直接叩く。200、image/jpeg。サーバーは正しい
- デプロイ済みのJavaScriptチャンクをダウンロードして開く。絶対URL化のコードが、APIオリジンの文字列ごと焼き込まれている。配信中のフロントも正しい
- ページに付く Content-Security-Policy の `img-src` には、バックエンドのオリジンを許可済み。CSPも正しい

全員にアリバイがある。
残るのは「curlにも自動テストにも見えず、ブラウザにだけ見える何か」である。

CORSを疑いたくなるところだが、`<img>` タグによる画像の読み込みは no-cors のリクエストで、そもそもCORSの検査対象ではない[^cors]。
となると、ブラウザが独自に強制するレスポンスヘッダの側だ。
画像URLのレスポンスヘッダを、今度は本文ではなくヘッダを目当てに眺めた[^head]。

```
HTTP/1.1 200 OK
Content-Type: image/jpeg
Cross-Origin-Resource-Policy: same-origin
```

CORSではない。
**CORP** である。

## ブラウザだけが強制するヘッダ

**Cross-Origin-Resource-Policy（CORP）** は、リソースの側が「自分をどのオリジンのページに埋め込んでよいか」を宣言するレスポンスヘッダである。
`same-origin` が付いていると、別オリジンのページの `<img>` から読み込まれたとき、ブラウザは応答を受け取ったうえで描画せずに捨てる。
Spectre 系の攻撃（クロスオリジンのリソースを自プロセスに読み込ませて盗み見る手口）への防御として導入された仕組みで、強制するのはブラウザだけだ。
curlはこのヘッダを素通しするし、fetch の cors モード通信も対象外なので、サーバーサイドの検証には一切映らない。
「curlでは200、ブラウザでは壊れる」という非対称は、このヘッダの仕様そのものだった。

では誰が付けたのか。
使っているWebフレームワーク Hono の `secureHeaders()` ミドルウェアが、デフォルトで `same-origin` を付けていた。
フロントとAPIを同一オリジンで配信していた頃には、正しく、かつ無害な既定値である。
別オリジン構成に移行した日に、この既定値の意味だけが静かに変わっていた。

## メディア配信だけを開ける

画像や動画は、別オリジンのフロントに埋め込まれることが正規の用途である。
だからメディア配信のレスポンスにだけ `Cross-Origin-Resource-Policy: cross-origin` を付ければよい。

ただし一つ引っかかった。
`secureHeaders()` はレスポンス確定後にヘッダを付けるため、ルート側で `cross-origin` を指定しても `same-origin` に上書きされてしまう。
そこでミドルウェア側のCORP付与を止め、メディア配信のルートで明示的に付ける形にした。

```ts
// ミドルウェア: CORPの一括付与をやめる
app.use(
  secureHeaders({
    crossOriginResourcePolicy: false,
    // ...他の設定はそのまま
  })
);

// メディア配信レスポンス: 埋め込みを明示的に許可
return new Response(data, {
  headers: {
    "Content-Type": contentType,
    "Cache-Control": "public, max-age=31536000, immutable",
    "Cross-Origin-Resource-Policy": "cross-origin",
  },
});
```

JSONを返すAPIからはCORPが消えることになるが、JSONは `<img>` にも `<script src>` にも意味のある形で埋め込めないので、実質的に失うものはない。

検証のやり方も一つ変えた。
ルート単体のユニットテストだけでなく、ローカルで実サーバーを起動し、ミドルウェアを通過した後の最終ヘッダに `cross-origin` が残ることを確かめてからデプロイした。
ミドルウェアの上書きは、ルート単体のテストには映らないからである。

## 「curlで通る」が保証する範囲

curlの200は、「サーバーが正しく応答している」ことしか保証しない。
その応答をページに埋め込めるかどうかは、ブラウザが別のルールで決めている。
今回の整理はこうだ。

- **CORS（Access-Control-Allow-Origin など）**：別オリジンからの「読み取り」をリクエスト側に許可する仕組み。fetch や XHR に効く
- **CORP（Cross-Origin-Resource-Policy）**：リソースの側が「埋め込まれてよい」オリジンを制限する仕組み。`<img>` や `<script>` に効く
- **CSP（Content-Security-Policy）**：ページの側が、読み込んでよいリソースの出所を制限する仕組み

同じ「画像だけ壊れる」症状は、CSPの `img-src` 不足でも起きる。
だから犯人当てを机上でやるより、開発者ツールのコンソールとネットワークタブを最初に開くのが結局いちばん速い。
私はサーバー側のアリバイ集めを三周してから、ようやくレスポンスヘッダに目を落とした。

セキュリティヘッダのミドルウェアを入れた日、構成はすべて同一オリジンで、あの既定値は正解だった。
構成を変えた日に既定値を読み直す習慣は、なかった。
CORPには会ったが、同じ一族の COOP と COEP にはまだ踏み込んでいない。
別オリジン構成を続ける限り、いつか会う気がしている。

[^cors]: CORSとCORPは向きが逆である。CORSは「リクエストした側のオリジンに、応答の中身を読ませてよいか」をサーバーが許可する仕組みで、CORPは「リソースを埋め込もうとするページのオリジン」をリソース側が制限する仕組み。`<img>` の表示はCORSの許可を必要としない代わりに、CORPの制限を受ける。

[^head]: 白状すると、最初のヘッダ確認は `curl -I`（HEADリクエスト）でやって401が返り、数分だけ別の迷路に入った。このAPIは認証ミドルウェアがGETのメディア配信だけを公開ルートとして通す作りで、HEADは対象外だったのが理由である。測定のつもりで打ったリクエストが、本番のリクエストと別物になっていることもある。
