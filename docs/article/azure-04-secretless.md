# Key VaultとマネージドIDとOIDCで、シークレットの平文を消す（結婚式クイズアプリをAzureに載せた話 4）

「結婚式クイズアプリをAzureに載せた話」の第4回です。

この回の作業を始めた動機は、きれいな設計思想ではなく事故でした。
DBの認証トークンを、エディタの画面共有経由で一度露出させてしまったんです。
ローテーションのために「トークンがどこに何箇所あるか」を数えたら、.env、GitHubのSecret、App Serviceの環境変数の3箇所。
3箇所を手で更新して回りながら、「置き場所、減らしたい…」と思ったのが始まりです。

やったことは二段階です。

1. アプリが使うシークレット（管理画面PIN、DBトークン）をKey Vaultに集約し、App ServiceはマネージドIDで取りに行く
2. デプロイに使っていた発行プロファイル（長寿命の認証情報）を廃止し、GitHub ActionsをOIDCで認証する

どちらも追加コストは実質0円でした。
Key Vaultは操作1万回あたり数円の課金だけで、月額の固定費はありません。

## Key Vault編

### 作成時の選択はRBACモード一択

Key Vault作成ウィザードの「アクセス構成」で、旧来のアクセスポリシー方式とRBAC方式を選びます。
新規で作るならRBAC（Azureロールベースのアクセス制御）でよいと思います。
以降の権限付与がすべて「アクセス制御 (IAM)」に統一されて、他のリソースと同じ考え方で扱えるようになります。

「消去保護」は無効のまま作りました。
有効にすると後から取り消せず、リソースを消しても90日間は完全削除できなくなります。
結婚式が終わったら片付ける前提のプロジェクトには合わない設定です。
ランサムウェア対策やコンプライアンス要件があるなら有効にしてください。

### 最初のハマり：作った本人がシークレットを登録できない

Vaultを作ってすぐシークレットを登録しようとすると、403で拒否されます。
サブスクリプションの所有者であっても、です。
「私が作ったのに！？」となりました。

RBACモードのKey Vaultでは、リソースの管理権限（作成・削除・設定変更）とデータの操作権限（シークレットの読み書き）が完全に分離されています。
所有者が持っているのは前者だけで、シークレットを登録するには、自分に**キー コンテナー シークレット責任者**（Key Vault Secrets Officer）ロールを明示的に割り当てる必要があります。
割り当て後も反映に1〜5分かかることがあり、403のまま数分待つ場面もありました。

もう一つ、日本語ポータル固有の問題があります。
IAMのロール検索で「key vault」と入力しても、目当てのロールがヒットしません。
主要なKey Vaultロールの表示名は「キー コンテナー シークレット責任者」「キー コンテナー シークレット ユーザー」と日本語化されていて、英語名では部分一致しないんですよね。
「シークレット」で検索するのが早いです。

### App Serviceへの配線

手順は3つで、コード変更はありません。

1. App Serviceの「ID」でシステム割り当てマネージドIDを有効化する
2. Key VaultのIAMで、そのIDに**キー コンテナー シークレット ユーザー**（読み取り専用）を割り当てる
3. App Serviceの環境変数の値を、平文からKey Vault参照に書き換える

```
ADMIN_PIN = @Microsoft.KeyVault(SecretUri=https://kv-quiz-prod.vault.azure.net/secrets/ADMIN-PIN/)
```

アプリから見ると普通の環境変数のままで、実体だけがKey Vaultから解決されます。
参照が解決できたかは、環境変数の一覧に付く緑のチェックマークで確認できます。
赤い×のときは、だいたいIAMの反映待ちかURIの書き間違いでした。

細かい仕様を2つ。
シークレット名にアンダースコアは使えない（ハイフンのみ）ので、環境変数名との対応はハイフン名で取ります。
SecretUriの末尾からバージョンのハッシュを消してスラッシュで終えると「常に最新バージョン」の参照になり、ローテーションがVault側の更新だけで済みます。
ただしバージョンなし参照はApp Service側で最大24時間キャッシュされるので、ローテーション後は再起動もセットにしてください。

### Key Vault参照で何が変わって、何が変わらないか

変わったのは、App Serviceの設定画面にシークレットの平文が表示されなくなったこと、アクセスがKey Vaultの監査ログに残ること、ローテーションが1箇所の更新で済むようになったことです。

変わらないこともあります。
App Serviceへのデプロイ権限を持つ人は、コードを差し替えれば環境変数（解決後の値）を読めます。
Key Vault参照は「見えなくする魔法」ではなくて、管理とローテーションと監査を1箇所に寄せる仕組み、と理解しておくのがよさそうです。

## OIDC編

### 発行プロファイルの何が問題か

それまでのデプロイは、App Serviceの発行プロファイル（ユーザー名とパスワードが入ったXML）をGitHubのSecretに貼る方式でした。
これは有効期限のない認証情報で、漏れたらリセットするまで誰でもデプロイできてしまいます。
GitHub Actionsには、この種の長寿命シークレットを置かずにクラウドへ認証する仕組み（OIDCフェデレーション）があるので、そちらに切り替えました。

### 仕組みと設定

流れはこうです。

1. GitHub Actionsがジョブ実行時に短命のOIDCトークンを発行する（リポジトリ名や環境名が含まれる）
2. Entra IDがそのトークンを検証し、事前に登録した条件（どのリポジトリのどの環境か）に一致すればAzureのアクセストークンを発行する
3. ワークフローはそのトークンでデプロイする

Azure側の設定は3つです。

- ユーザー割り当てマネージドIDを作る（アプリ登録でもよいですが、こちらのほうが管理が単純でした）
- そのIDに「フェデレーション資格情報」を追加する。ウィザードにGitHub Actions用のテンプレートがあり、組織・リポジトリ・エンティティ（今回は環境 `production`）を入れるだけ
- App ServiceのIAMで、そのIDに**Webサイト共同作成者**ロールを割り当てる

GitHub側は、ワークフローに `permissions: id-token: write` を足して、`azure/login@v2` にclient-id、tenant-id、subscription-idを渡します。
この3つのIDはシークレットではない（漏れても、フェデレーション条件を満たすワークフローしか認証できない）ので、SecretsではなくVariablesに置きました。

```yaml
    permissions:
      id-token: write
      contents: read
    steps:
      - uses: azure/login@v2
        with:
          client-id: ${{ vars.AZURE_CLIENT_ID }}
          tenant-id: ${{ vars.AZURE_TENANT_ID }}
          subscription-id: ${{ vars.AZURE_SUBSCRIPTION_ID }}
      - uses: azure/webapps-deploy@v3
        with:
          app-name: ${{ vars.AZURE_WEBAPP_NAME }}
          package: deploy-package.zip
          # publish-profile は渡さない
```

切り替え後、発行プロファイルのSecretを削除して、App ServiceのSCM基本認証も無効に戻しました。
その状態でデプロイが通ることを確認して完了です。
デプロイ経路に、盗まれて困る長寿命の認証情報が存在しなくなりました。
気持ちいい！

### 本番だけOIDCにした

OIDC化は本番環境だけに適用して、dev環境は発行プロファイルのまま残しています。
フェデレーション資格情報の登録先をどう分けるか整理する前に、本番側を先に固めたかったのが理由で、ワークフローは環境名で認証方式を分岐しています。
結果として1本のワークフローに新旧両方式が並んでいて、見比べられる教材としては悪くないかなと思っています。
devも揃えるのは宿題です…

## さいごに

この回でいちばん頭に残ったのは、この2つでした。

- Key VaultのRBACで最初に403を食らったのは、**管理プレーン**（リソースを作る・消す権限）と**データプレーン**（中身を読む・書く権限）が別物だから
- OIDCで発行プロファイルを消せたのは、「長寿命の秘密を保管して照合する」認証を「短命のトークンをその場で検証する」認証に置き換えたから

どちらもAzureのドキュメントで名前だけは知っていた概念でしたが、手を動かして403に怒られて、やっと自分の言葉になった気がします。
