# 画像をBlob Storageへ、監視をApplication Insightsへ（結婚式クイズアプリをAzureに載せた話 5）

「結婚式クイズアプリをAzureに載せた話」、いよいよ最終回です。
残っていた課題は2つ。
アップロード画像（クイズの問題画像とゲストの自撮りアイコン）がApp Serviceのローカルディスク頼みであることと、本番の監視が何もないことです。

## Blob Storage移行

### ストレージアダプタを1枚挟む

移行前のコードは、ルートハンドラが `fs` を直接呼んでいました。
これをストレージドライバのインターフェースに置き換えて、実装をローカルディスクとBlobの2種類にしました。

```ts
interface StorageDriver {
  save(filename: string, data: Buffer, metadata?: Record<string, string>): Promise<void>;
  read(filename: string): Promise<Buffer | null>;  // null = not found
  delete(filename: string): Promise<void>;         // 冪等
  totalSize(): Promise<number>;                    // 容量上限チェック用
}
```

選択は環境変数 `STORAGE_DRIVER` でやります。
未設定ならローカルディスクなので、既存の開発環境とテストは無変更で動きます。
未知の値のときは起動時に例外を投げるようにしました。
タイポで意図せずローカル保存に落ちると、データが2箇所に分かれてから気付くことになるからです。それは怖すぎる…

Blob実装のユニットテストは、SDKをモックする代わりに、コンストラクタでContainerClient相当のオブジェクトを受け取る設計にして、テストではフェイクを注入しました。
SDKの型のうち使うメソッドだけを構造的部分型で切り出すと、フェイクが数十行で書けます。

### 認証はマネージドID

本番のBlobアクセスに接続文字列は使わず、第4回で有効化したApp ServiceのマネージドIDをそのまま使いました。

- コード側: `new ContainerClient(url, new DefaultAzureCredential())`
- Azure側: ストレージアカウントのIAMで、App ServiceのIDに**ストレージ BLOB データ共同作成者**を割り当てる

ロール名は似たものがあるので要注意です。
「ストレージ アカウント共同作成者」は管理プレーンのロールで、割り当ててもBlobの読み書きはできません（第4回で覚えた管理プレーン/データプレーンの話、ここでも出てきます）。
データプレーンのロールは、名前に「BLOB データ」が入っています。

ストレージアカウント作成時の注意は1つで、冗長性の既定がGRS（geo冗長）になっていることがあります。
このアプリの画像に地理冗長は不要なので、LRSを明示的に選びました。
保存量は式全体でも数百MBに収まる想定で、月のストレージ費用は数円です。

### フォルダ分けは、ファイル名にコンテキストを埋め込んで実現

最初の実装はコンテナ直下にフラットに保存していて、ポータルで見たとき「どのblobがどのクイズの何なのか」が名前から判別できませんでした。
ファイル名はセキュリティ上ランダム（推測不可）にしたいので、元のファイル名をそのまま使うわけにもいきません。

採用したのは、ランダム名の接頭辞に種別とID（クイズID、ルームコード）を符号化して、保存時にストレージキーへ展開する方式です。

| 種別 | ファイル名 | Blob上のキー |
|---|---|---|
| 問題画像 | `q_{quizId}_{ランダム}.jpg` | `questions/{quizId}/...` |
| 選択肢画像 | `c_{quizId}_{ランダム}.jpg` | `choices/{quizId}/...` |
| 自撮り | `selfie_{roomCode}_{ランダム}.jpg` | `selfies/{roomCode}/...` |

URLは従来どおり1階層（`/api/media/ファイル名`）のままで、ルーティングもDBのスキーマも変えていません。
配信・削除の入り口で、ファイル名からキーを導出する関数を1つ通すだけです。
接頭辞のない旧形式の名前はコンテナ直下として扱い、後方互換にしました。

あわせて、blobのメタデータに元のファイル名・種別・紐付け先IDを記録しました。
ポータルでblobを開けば「元は何というファイルで、どのクイズのものか」が確認できます。
1点だけ制約があって、Azureのblobメタデータの値はASCII限定なので、日本語のファイル名はURLエンコードして格納しています。

### 移行してすぐ、画像が表示されなくなった

デプロイ後の動作確認で、アップロードは成功するのに表示だけが失敗する問題を踏みました。
原因は2層あって、フロントが相対URLで画像を参照していたこと（別オリジン構成では解決先が違う）と、セキュリティヘッダのミドルウェアが付ける `Cross-Origin-Resource-Policy: same-origin` がブラウザの画像埋め込みをブロックしていたことでした。
後者はcurlでもテストでも検出できない種類の問題で、切り分けに2時間かかりました…
経緯と対処は分量があるので、番外編に分けています。

→ 番外編: 「curlだと200なのにブラウザだと画像が表示されない」の原因がCORPだった話（Qiita投稿後にここへリンクを貼る）

## Application Insights

### 有効化はコード変更ゼロで始められる

App Serviceのポータルに「Application Insights」の項目があり、有効化するとエージェントが自動計装でHTTPリクエストと外部依存の計測を始めてくれます。
Node 22 + ESMのアプリですが、コード側の変更なしで計測が入りました。
料金はLog Analyticsへの取込5GB/月まで無料で、この規模のアプリではまず使い切れません。

計測が入っていることの確認には、Live Metricsが手軽です。
画面を開いたままアプリを操作すると、リクエストが数秒遅れでグラフに流れてきます。
ちょっと感動しました。
本番当日は、この画面を進行席のPCに出しておく運用にしています。
障害が起きたとき、最初の30秒で「そもそもサーバーに届いているのか」を判別できる手段があるのは大きいです。

### KQLで最初の1本を書く

溜まったテレメトリはKQL（Kusto Query Language）で集計できます。
最初に書いたのは「5分ごとのリクエスト数とp95応答時間をステータスコード別に見る」クエリです。

```kusto
requests
| where timestamp > ago(1h)
| summarize count_ = count(), p95_ms = percentile(duration, 95) by bin(timestamp, 5m), resultCode
| order by timestamp desc
```

細かい話ですが、列名に日本語（`件数 = count()`）を使うと構文エラーになります。
使いたい場合は `['件数'] = count()` とブラケットで囲みます。（地味に数分ハマりました）

このクエリで取れた実測値が、性能の当たりを付けるのに役立ちました。
DBを触らないヘルスチェックのp95が9ms、Tursoへの読み取りを含むAPIのp95が279ms。
差分がほぼそのままDB往復のコストなので、「遅くなったらまずDB依存を見る」という切り分けの基準を数字で持てました。

### アラートは見送り、既定の検出に任せた

5xxの増加でメール通知するアラートルールも作りかけたのですが、メトリックアラートは1本あたり月十数円かかります。
金額としては誤差でも、このプロジェクトは「月0円」を縛りにしているので見送りました（ルールはルール）。
代わりになるものとして、Application Insightsには既定で有効なスマート検出（失敗率の異常上昇を自動検知して通知する機能）が無料で付いています。
常時監視が必要なサービスならアラートルールを作るべきで、ここは要件との相談だと思います。

## シリーズのまとめ

5回で書いた内容を、費用の表で締めます。

| 項目 | 月額 |
|---|---|
| Static Web Apps（Free） | 0円 |
| App Service F1（待機時） | 0円 |
| App Service B1（リハーサル・本番日のみ） | 約65円/日 |
| Turso（Free） | 0円 |
| Key Vault | 実質0円 |
| Blob Storage（LRS） | 数円 |
| Application Insights（5GB/月まで） | 0円 |

個人開発の規模でも、マネージドID、Key Vault、OIDC、可観測性といった「業務のAzureで使う道具」を一通り実地で触れました。
ドキュメントを読むだけだと、RBACの403にも、CIFSの上で動かないWALにも、ブラウザだけが強制するヘッダにも会えないんですよね。
無料枠は、そういうものに安全にぶつかりに行く練習場として十分に広かったです。
