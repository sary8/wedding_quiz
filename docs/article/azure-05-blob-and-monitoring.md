# 画像をBlob Storageへ、監視をApplication Insightsへ（結婚式クイズアプリをAzureに載せた話 5）

シリーズ最終回である。
残っていた課題は二つ。
アップロード画像（クイズの問題画像とゲストの自撮りアイコン）がApp Serviceのローカルディスク頼みであることと、本番の監視が何もないことである。

## Blob Storage移行

### ストレージアダプタを1枚挟む

移行前のコードは、ルートハンドラが `fs` を直接呼んでいた。
これをストレージドライバのインターフェースに置き換え、実装をローカルディスクとBlobの2種類にした。

```ts
interface StorageDriver {
  save(filename: string, data: Buffer, metadata?: Record<string, string>): Promise<void>;
  read(filename: string): Promise<Buffer | null>;  // null = not found
  delete(filename: string): Promise<void>;         // 冪等
  totalSize(): Promise<number>;                    // 容量上限チェック用
}
```

選択は環境変数 `STORAGE_DRIVER` で行う。
未設定ならローカルディスクなので、既存の開発環境とテストは無変更で動く。
未知の値のときは起動時に例外を投げる。
タイポで意図せずローカル保存に落ちると、データが2箇所に分かれてから気付くことになるからである。

Blob実装のユニットテストは、SDKをモックする代わりに、コンストラクタでContainerClient相当のオブジェクトを受け取る設計にして、テストではフェイクを注入した。
SDKの型のうち使うメソッドだけを構造的部分型で切り出すと、フェイクが数十行で書ける。

### 認証はマネージドID

本番のBlobアクセスに接続文字列は使わず、第4回で有効化したApp ServiceのマネージドIDをそのまま使った。

- コード側: `new ContainerClient(url, new DefaultAzureCredential())`
- Azure側: ストレージアカウントのIAMで、App ServiceのIDに**ストレージ BLOB データ共同作成者**を割り当てる

ロール名は似たものがあるので注意する。
「ストレージ アカウント共同作成者」は管理プレーンのロールで、割り当ててもBlobの読み書きはできない。
データプレーンのロールは名前に「BLOB データ」が入っている。

ストレージアカウント作成時の注意は一つで、冗長性の既定がGRS（geo冗長）になっていることがある。
このアプリの画像に地理冗長は不要なので、LRSを明示的に選んだ。
保存量は式全体でも数百MBに収まる想定で、月のストレージ費用は数円である。

### フォルダ分けは、ファイル名にコンテキストを埋め込んで実現した

最初の実装はコンテナ直下にフラットに保存していて、ポータルで見たとき「どのblobがどのクイズの何なのか」が名前から判別できなかった。
ファイル名はセキュリティ上ランダム（推測不可）にしたいので、元のファイル名をそのまま使う解決は取れない。

採用したのは、ランダム名の接頭辞に種別とID（クイズID、ルームコード）を符号化し、保存時にストレージキーへ展開する方式である。

| 種別 | ファイル名 | Blob上のキー |
|---|---|---|
| 問題画像 | `q_{quizId}_{ランダム}.jpg` | `questions/{quizId}/...` |
| 選択肢画像 | `c_{quizId}_{ランダム}.jpg` | `choices/{quizId}/...` |
| 自撮り | `selfie_{roomCode}_{ランダム}.jpg` | `selfies/{roomCode}/...` |

URLは従来どおり1階層（`/api/media/ファイル名`）のままで、ルーティングもDBのスキーマも変えていない。
配信・削除の入り口でファイル名からキーを導出する関数を1つ通すだけである。
接頭辞のない旧形式の名前はコンテナ直下として扱い、後方互換にした。

あわせて、blobのメタデータに元のファイル名・種別・紐付け先IDを記録した。
ポータルでblobを開けば「元は何というファイルで、どのクイズのものか」が確認できる。
一点だけ制約があり、Azureのblobメタデータの値はASCII限定なので、日本語のファイル名はURLエンコードして格納している。

### 移行してすぐ、画像が表示されなくなった

デプロイ後の動作確認で、アップロードは成功するのに表示だけが失敗する問題を踏んだ。
原因は2層あって、フロントが相対URLで画像を参照していたこと（別オリジン構成では解決先が違う）と、セキュリティヘッダのミドルウェアが付ける `Cross-Origin-Resource-Policy: same-origin` がブラウザの画像埋め込みをブロックしていたことである。
後者はcurlでもテストでも検出できない種類の問題で、切り分けに2時間かかった。
経緯と対処は分量があるので、別記事「curlでは200が返るのにブラウザで画像が壊れるとき、Cross-Origin-Resource-Policy を疑う」に分けてある。

## Application Insights

### 有効化はコード変更ゼロで始められる

App Serviceのポータルに「Application Insights」の項目があり、有効化するとエージェントが自動計装でHTTPリクエストと外部依存の計測を始める。
Node 22 + ESMのアプリだが、コード側の変更なしで計測が入った。
料金はLog Analyticsへの取込5GB/月まで無料で、この規模のアプリでは使い切れない。

計測が入っていることの確認には、Live Metricsが手軽である。
画面を開いたままアプリを操作すると、リクエストが数秒遅れでグラフに流れる。
本番当日は、この画面を進行席のPCに出しておく運用にした。
障害が起きたとき、最初の30秒で「サーバーに届いているのか」を判別できる手段があるのは大きい。

### KQLで最初の1本を書く

溜まったテレメトリはKQLで集計できる。
最初に書いたのは「5分ごとのリクエスト数とp95応答時間をステータスコード別に見る」クエリである。

```kusto
requests
| where timestamp > ago(1h)
| summarize count_ = count(), p95_ms = percentile(duration, 95) by bin(timestamp, 5m), resultCode
| order by timestamp desc
```

細かい話だが、列名に日本語（`件数 = count()`）を使うと構文エラーになる。
使いたい場合は `['件数'] = count()` とブラケットで囲む。

このクエリで取れた実測値が、性能の当たりを付けるのに役立った。
DBを触らないヘルスチェックのp95が9ms、Tursoへの読み取りを含むAPIのp95が279msである。
差分がほぼそのままDB往復のコストで、「遅くなったらまずDB依存を見る」という切り分けの基準が数字で持てた。

### アラートは見送り、既定の検出に任せた

5xxの増加でメール通知するアラートルールも作りかけたが、メトリックアラートは1本あたり月十数円かかる。
金額としては誤差でも、このプロジェクトは「月0円」を縛りにしていたので見送った。
代わりになるものとして、Application Insightsには既定で有効なスマート検出（失敗率の異常上昇を自動検知して通知する機能）が無料で付いている。
常時監視が必要なサービスならアラートルールを作るべきで、これは要件との相談である。

## シリーズのまとめ

5回で書いた内容を、費用の表で締める。

| 項目 | 月額 |
|---|---|
| Static Web Apps（Free） | 0円 |
| App Service F1（待機時） | 0円 |
| App Service B1（リハーサル・本番日のみ） | 約65円/日 |
| Turso（Free） | 0円 |
| Key Vault | 実質0円 |
| Blob Storage（LRS） | 数円 |
| Application Insights（5GB/月まで） | 0円 |

個人開発の規模でも、マネージドID、Key Vault、OIDC、可観測性といった「業務のAzureで使う道具」を一通り実地で触れた。
ドキュメントを読むだけでは、RBACの403にも、CIFSの上で動かないWALにも、ブラウザだけが強制するヘッダにも会えない。
無料枠は、それらに安全に会うための練習場として十分に広かった。

残る作業は、本番前の負荷テストである。
Azure Load Testingに月50仮想ユーザー時間の無料枠があり、100人の一斉回答をB1に対して再現するつもりでいる。
結果は本番（結婚式）が終わったら追記したい。
