# 月0円でWebSocketアプリをAzureに本番構成で載せる（結婚式クイズアプリをAzureに載せた話 1）

結婚式の余興用に、リアルタイムクイズアプリを個人開発しています。
React + Vite / Hono + Socket.io の構成で、ゲスト100人がスマホから一斉に回答する、本番は結婚式当日の1日だけ、というちょっと変わったアプリです。

3年目のエンジニアとして業務でAzureに触る機会が増えてきたこともあり、このアプリを教材にして、本番相当の構成を自分の手で組んでみることにしました。
先に白状しておくと、会社のサブスクリプションではなく自腹です。
なので予算は月0円、使うのは無料枠だけ、と決めました。
「無料枠だけで本番構成なんて組めるのか…？」と半信半疑で始めましたが、結論、組めました。
待機時は月額0円、本番当日だけ約65円です。

本記事はシリーズの第1回で、全体構成と無料枠の考え方を書きます。
私と同じように「Azureを頑張りたいけど、業務で触れる範囲は限られている」「個人でやるなら絶対に無料がいい」という方の参考になればうれしいです。

1. 月0円でWebSocketアプリをAzureに本番構成で載せる（この記事）
2. SQLiteのままApp Serviceに載せてはいけない理由とTurso移行
3. Azure初デプロイで詰まった10のこと
4. Key VaultとマネージドIDとOIDCで、シークレットの平文を消す
5. 画像をBlob Storageへ、監視をApplication Insightsへ

## 最初の壁は「無料でWebSocket」

静的サイトを無料で公開する方法はいくらでもあります。
問題はバックエンドでした。
このアプリはSocket.ioで100人が常時接続するので、WebSocketが使えない無料枠は選択肢にならないんですよね。

最初の候補は、App Serviceの無料プランF1でした。
調べてみると、F1には本番投入を諦めるのに十分な制約がありました。

| 制約 | F1（無料） | B1（約2,000円/月） |
|---|---|---|
| WebSocket同時接続数 | **5本/インスタンス** | 350本 |
| CPUクォータ | **60分/日**（超過すると翌日まで応答停止） | なし |
| Always On | 不可（アイドル20分でアンロード） | 可 |

同時5接続では、ゲスト100人どころかリハーサルもできません。
CPUクォータも致命的で、イベントの途中で60分を使い切ると、翌日まで503が返り続けます。
復旧手段がないまま結婚式が終わる。想像したくなさすぎる…

Azure Container Appsの無料枠（月18万vCPU秒）も検討しましたが、こちらはコンテナのファイルシステムがエフェメラルで、当時ローカルSQLiteだったDBがレプリカの再起動で消えます。
Cosmos DBの無料枠（1,000 RU/s）でDBごと置き換える案も、100人の一斉回答が書き込みバーストになってちょうど1,000 RU/sの上限に当たる計算になり、しかもリレーショナルなスキーマをNoSQLに書き直す工数が見合いませんでした。

## 発想の転換：F1とB1を行き来する

答えはプランの中間にありました。
App Serviceは時間課金で、F1とB1は同一プランのスケール変更で行き来できます。
設定もデプロイ済みのコードもそのまま残ります。

- 普段: F1に落としておく（0円。1人での動作確認は5接続以内で可能）
- リハーサルと本番当日: B1に上げる（約65円/日）
- 終わったら: F1に戻す

「本番は1日だけ」という要件なら、この運用で月額はほぼ0円になります。
逆に、常時ユーザーがいるサービスだとこの手は使えないので、B1の約2,000円/月を最初から見込むことになります。

## 最終的な構成

2日かけて組み上げた構成がこちらです。

| 役割 | サービス | プラン / 月額 |
|---|---|---|
| フロントエンド（React SPA） | Azure Static Web Apps | Free / 0円 |
| バックエンド（Hono + Socket.io） | Azure App Service（Linux, Node 22） | F1⇔B1 / 0円 + 稼働日×約65円 |
| DB | Turso（ホスト型libSQL・外部SaaS） | Free / 0円 |
| シークレット管理 | Azure Key Vault | 実質0円（操作課金のみ） |
| アップロード画像 | Azure Blob Storage（LRS） | 数円 |
| 監視 | Application Insights | 取込5GB/月まで0円 |
| CI/CD | GitHub Actions | publicリポジトリのため0円 |

DBだけAzureじゃないのには理由があって、第2回で書きます。
短く言うと、App ServiceのファイルシステムにSQLiteを置けない事情があり、置き換え先としてコード変更が最小のTursoを選びました。

（構成図はリポジトリの docs/prod-architecture.drawio にあります。Qiita投稿時はエクスポートした画像をここに貼る）

通信の要点は3つです。

- ブラウザは Static Web Apps から静的ファイルを受け取り、APIとSocket.ioは App Service へ直接つなぐ（別オリジン構成）
- App Service から Key Vault と Blob Storage へのアクセスは、パスワードではなくマネージドIDで認証する（第4回・第5回）
- デプロイは GitHub Actions から。本番へは承認ボタンを押さない限り出ない

## dev/prod分離も0円でできた

本番環境を作る前に、同じ構成の検証環境（dev）を先に作りました。
理由は単純で、ポータル操作を間違えても壊れるのがdevで済むからです。
実際、初回はdevで価格プランの選択を間違えかけました（第3回で書きます）。危ない危ない…

追加コストは0円でした。

- Static Web Apps Free は複数作れる
- App Service F1 も複数作れる（devは常時F1のまま）
- Turso Free はDBを複数持てる

環境の切り替えはGitHub Environments（`development` / `production`）でやっています。
環境ごとに同名のSecretsとVariablesを持たせて、ワークフローの `environment:` で参照先が切り替わる仕組みです。
mainへのpushでdevに自動デプロイ、本番はActionsの手動実行と承認ゲート（Required reviewers）を通します。
publicリポジトリなら、Environmentsの保護ルールも無料で使えます。

この分離の目的は、「うっかりmainにマージしたら本番が変わってしまう」状態を作らないことです。
結婚式の前日に本番へ出るのは、自分が承認ボタンを押したときだけ、になりました。

## リージョンの注意：Japan Eastで作れないことがある

リソースはすべてJapan Westに置きました。
最初はJapan Eastを選んだのですが、App Serviceの作成が「Current Limit (Total VMs): 0」というエラーで通りませんでした。
新しめの個人サブスクリプションだと、人気リージョンのVMクォータが0に設定されていることがあるようです。
無料のF1でもこのクォータの対象になります。
クォータ引き上げの申請もできますが、急ぐならリージョンを変えたほうが早いです。
Japan WestとJapan Eastのレイテンシ差は数ミリ秒で、このアプリの用途では測っても体感できませんでした。

## この構成にしなかったもの

検討して見送った選択肢も書いておきます。

- **Azure Container Apps**: スケールtoゼロ前提の無料枠は魅力だが、常時1レプリカを維持すると無料枠を超える。WebSocketの常時接続と相性が悪い
- **Azure SQL Database の無料オファー**: 使っているORM（Drizzle）がSQL Server方言に対応しておらず、クエリ層の書き直しが必要だった
- **Azure Web PubSub**: 無料枠が同時20接続で、100人の要件に届かない

見送りの基準はどれも同じで、「無料枠の制約が要件の数字と衝突するか」「既存コードの書き直し量が見合うか」の2点でした。
サービスの善し悪しではなく、要件との掛け算で決まるんだなと実感しています。
