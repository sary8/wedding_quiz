## 1. システム構成

```
参加者スマホ（最大100人）
      ↕ Socket.io / HTTP
    バックエンド API（Node.js + Hono）
      ↕ SQLite
    フロントエンド（React + Vite）
      ↙ ホストPC    ↘ プロジェクター
```

| 役割 | URL（本番） | URL（ローカル開発） |
|------|-------------|-------------------|
| フロントエンド | Azure Static Web Apps の URL | http://localhost:5174 |
| バックエンド API | Azure App Service の URL | http://localhost:3001 |

## 2. 事前準備

### 2-1. 必要なもの

- **ホスト用PC**（司会者が操作）— ブラウザが動けば何でも可
- **プロジェクター + HDMI ケーブル**（または大型モニター）
- **Wi-Fi ルーター**（参加者全員が接続できるもの）
- **参加者スマホ**（QRコード読み取り可能なブラウザ搭載端末）

### 2-2. ネットワーク要件

- ホストPC・プロジェクターPC・参加者スマホが **同一ネットワーク**（またはインターネット経由）で接続できること
- バックエンドのポート **3001**（開発時）または Azure URL（本番）が全端末からアクセス可能なこと

### 2-3. ブラウザ推奨

| 端末 | 推奨ブラウザ |
|------|-------------|
| ホストPC | Chrome 最新版 |
| プロジェクター | Chrome 最新版 |
| 参加者スマホ | Chrome / Safari 最新版 |

## 3. クイズ作成（SetupPage）

### アクセス方法

```
http://<サーバーURL>/host/setup
```

> **PIN認証**: 環境変数 `ADMIN_PIN` が設定されている場合、初回アクセス時にPIN入力画面が表示されます。正しいPINを入力するとセッションが開始され、以降は認証不要です。

### 手順

#### Step 1: クイズを作成する

1. 「タイトル」欄にクイズ名を入力（例: `新郎新婦クイズ`）
2. 「クイズを作成」ボタンをクリック
3. 作成したクイズが画面下部の一覧に表示される

#### Step 2: 問題を追加する

各問題について以下の項目を入力：

| 項目 | 説明 | 必須 |
|------|------|------|
| 問題文 | 参加者に表示する質問文 | ◎ |
| 選択肢1〜4 | 4つの回答候補 | ◎ |
| 正解番号 | 1〜4のいずれか | ◎ |
| 制限時間 | 5〜120秒（デフォルト20秒） | ◎ |
| 画像 | 問題に表示する画像（任意） | — |

**画像添付の手順：**
1. 「画像を選択」ボタンをクリックしてファイルを選択（JPG・PNG・GIF・WebP、最大5MB）
2. サムネイルが即座に表示される（バックグラウンドでアップロード中）
3. アップロード完了後、問題文の横に「✓ 完了」バッジが表示される
4. 「追加」ボタンが活性化したら問題を保存できる
5. 削除したい場合は「×」ボタンをクリック

**問題の並べ替え：**
- ドラッグ&ドロップで問題の順番を変更できる（グリップアイコンを掴んで移動）
- ↑↓ボタンでも1つずつ移動可能

**問題バンク（テンプレート）：**
- ダッシュボードの「問題ライブラリ」からテンプレート問題を管理できる
- よく使う問題をテンプレートとして保存し、別のクイズにインポート可能
- 過去のクイズで使った問題も一覧で確認できる

#### Step 2.5: プレビューで確認する

1. 「プレビュー」ボタンをクリックすると参加者視点で問題を確認できる
2. 正解の選択肢がハイライト表示される
3. 前へ/次へボタンで問題を切り替え

#### Step 2.7: チーム設定

チーム戦は**常に有効**です（個人戦と並行して、チームスコアが自動集計されます）。

幹事はチームの「数」を設定します：

1. クイズ設定カード内の「チーム数」を選択する（2〜26個）
2. 参加者が自分で A〜Z のレター名チームから選んで参加します（例: チームA、チームB…）
3. 「チームを保存」ボタンをクリック

> **注意**: チーム数を変更すると、既存参加者のチーム割り当てがリセットされます。ロビーを開く前に確定してください。

#### Step 3: ロビーを開く

1. 問題をすべて追加したら「ロビーを開く」ボタンをクリック
2. 自動的にホスト操作画面（`/host/:roomCode`）へ遷移する

**リハーサルモード：**
- 「リハーサル」ボタンをクリックすると、リハーサルモードで開始
- 画面上部に黄色バナー「リハーサルモード」が表示される
- 最終問題後に5秒待って自動的にリプレイ（ロビーに戻る）
- 本番前の動作確認に使用

> **注意**: ホスト認証情報は sessionStorage に保存されます。
> 通常は SetupPage から遷移すればそのまま引き継がれます。

## 4. 当日の進行手順

### 4-1. サーバー起動

**ローカル開発環境の場合：**

ターミナルを2つ開いて、それぞれ実行：

```bash
# ターミナル1：バックエンド
cd src/backend
npm run dev
# → ポート3001で起動

# ターミナル2：フロントエンド
cd src/frontend
npm run dev
# → http://localhost:5174 で起動
```

**本番環境（Azure）の場合：**

Azure Portal でバックエンドの App Service を起動済みであることを確認。

---

### 4-2. ロビー開設〜参加者受付

1. ホストPC でブラウザを開き、`/host/setup` へアクセス
2. クイズを選択または新規作成して「ロビーを開く」
3. ホスト操作画面に **QRコード** と **ルームコード（6桁数字）** が表示される
4. 参加者にQRコードを読み取らせる（または `/play` のURLとルームコードを口頭で案内）
5. 参加者一覧にニックネームが表示されていくのを確認

---

### 4-3. プロジェクター接続

プロジェクターには **操作ボタンなしの表示専用画面** を使います。

1. プロジェクターに接続したPCのブラウザで以下のURLを開く：

```
http://<サーバーURL>/host/<ルームコード>/screen
```

例: `http://localhost:5174/host/ABCDEF/screen`

> **ヒント**: ホスト操作画面のロビー画面の下部に「プロジェクター用URL」が表示されているのでコピー＆ペーストしてください。

2. プロジェクター画面はホスト操作画面と **自動同期** します
3. ホストが操作するたびに画面が切り替わります

---

### 4-4. ゲーム開始〜問題進行

#### ゲーム開始

1. 参加者が揃ったら「ゲーム開始」ボタンをクリック
2. 5秒のカウントダウンが表示される（5→4→3→2→1）
3. カウントダウン終了後、自動的に最初の問題が配信される

#### 問題の進め方

各問題で以下の流れを繰り返す：

```
[次の問題] → 問題表示・カウントダウン開始
    ↓ (時間内に参加者が回答)
[回答を締め切る] または 時間切れで自動締め切り
    ↓
正解と回答分布を確認
    ↓
[ランキングを表示] → 上位10名のバーアニメーション
    ↓
[次の問題] or [最終結果発表]（最終問題の場合）
```

#### 操作ボタン一覧（問題フェーズ）

| ボタン | タイミング | 説明 |
|--------|----------|------|
| 次の問題 | 準備完了時 | 次の問題を全員に配信 |
| 回答を締め切る | 問題表示中 | 手動で早期締め切り |
| ランキングを表示 | 結果確認後 | 上位10名のバーグラフを表示 |
| 最終結果発表 | 全問終了後 | オールスター感謝祭式の発表 |

---

### 4-5. ランキング〜最終発表

#### ランキング画面

- 10名ずつページネーション表示（ホストが「前の10人」「次の10人」ボタンで切り替え）
- 左からバーが伸びるアニメーションでスコアを表示（満点に対する割合で長さが決まる）
- スコアが高い順に表示
- 「チームランキングを表示」ボタンでチーム別ランキングに切り替え可能（チーム名・合計スコア・メンバー数）。「個人ランキングに戻る」ボタンで個人に戻れる
- 参加者のスマホには自分の順位と上位5名のミニランキングが表示される

#### 終盤ブラックアウト（最後の5問）

緊張感を高めるため、**全問題のうち最後の5問**は個人順位・チーム順位がともに非表示になります。

- 参加者スマホの「現在順位」表示も非表示になります（正解/不正解・獲得ポイントは引き続き表示されます）
- 結果確認後はランキング画面をスキップして次の問題へ進みます（ホストの操作は「次の問題」のみ。最終問題の場合は「最終結果発表」のみ）
- 総問題数が5以下の場合は全問が対象（順位は最終結果発表まで一切表示されません）
- **最終結果発表（final）では順位を含むすべての情報が通常通り表示されます**

#### 最終結果発表

- 全問題終了後「最終結果発表」ボタンをクリック
- まずチームランキングが発表される（下位から順に表示、1位チームに紙吹雪）
- その後、10位から1位へカウントダウン形式で個人発表（オールスター感謝祭スタイル）
- 自動進行（操作不要）
- 参加者スマホには自分の最終順位・正答数・平均回答速度・チーム成績が表示される

#### 集合写真フェーズ

- Top3発表後、5秒で自動的に集合写真フェーズへ遷移する
- 全参加者のアバターがスクリーンセーバーのように画面内を浮遊する
- 自撮り写真を撮った参加者は顔写真が表示される

#### もう一度プレイ（リプレイ）

- 集合写真フェーズで「もう一度プレイ」ボタンが表示される
- クリックすると全参加者のスコアと回答データがリセットされ、ロビーに戻る
- 参加者のニックネーム・自撮りはそのまま維持される
- リハーサル後の本番開始や、別のクイズセットで再プレイしたいときに使用

#### ゲーム終了

- 集合写真フェーズで「ゲーム終了」ボタンをクリックすると、全参加者にサンキュースクリーンが表示される
- 参加者のアバターが浮遊するアニメーションと「ご参加ありがとうございました！」メッセージが表示される
- ホストには「管理画面に戻る」ボタンと「ルームを削除」ボタンが表示される
- 「ルームを削除」は2段階確認（「ルームを削除」→「本当に削除する」）で、クイズ・参加者・メディアを完全に削除する

#### ルームの自動クリーンアップ

- ゲーム終了（`finished`）から **1時間** 経過したルームは自動的に削除される
- サーバー起動時に1回 + 10分間隔で定期チェック
- 削除対象はDB上のクイズデータとアップロード済みメディアファイル

### 4-6. ホスト復旧

ゲーム進行中にホストのブラウザを閉じてしまった場合：

1. 同じURL（`/host/:roomCode?quizId=...`）でブラウザを再度開く
2. 「ゲームを再開」画面が表示される
3. 「次の問題を配信」または「ランキング表示」ボタンで続行できる
4. 参加者やプロジェクターの画面はそのまま維持される

> **注意**: サーバー自体が再起動した場合はゲーム状態がリセットされるため、最初からやり直す必要があります。

### 4-7. BGMについて

ホスト操作画面とプロジェクター画面では、フェーズに応じたBGMが自動的に切り替わります。

#### BGMファイルの準備

`public/audio/` フォルダに以下のMP3ファイルを配置してください：

| ファイル名 | 使用フェーズ |
|-----------|-------------|
| `lobby.mp3` | ロビー（参加者受付中） |
| `question.mp3` | 問題出題中 |
| `results.mp3` | 結果・ランキング・最終発表 |

- MP3ファイルはループ再生されます
- ファイルが未配置の場合でもエラーにはなりません（無音で動作）
- 著作権フリーのBGMをご使用ください

#### BGMコントロール

- 画面右下のスピーカーアイコンをクリックするとコントロールパネルが開く
- 音量スライダーとミュートボタンで調整可能
- 設定はブラウザに保存され、次回も維持される

---

### 4-8. 効果音について

ホスト操作画面とプロジェクター画面では、各フェーズで効果音が自動再生されます。

| タイミング | 効果音 |
|----------|--------|
| 参加者が参加したとき | 上昇チャイム |
| 問題が配信されたとき | 緊張感のあるパターン音 |
| カウントダウン残り5秒以下 | カウント音（毎秒） |
| 回答締め切り | ブザー音 |
| 結果発表 | 上昇音 |
| ランキング表示 | アルペジオ |
| ゲーム終了（最終発表開始） | ドラムロール |
| Top3 スポットライト | ランク別ファンファーレ |

- **参加者のスマホでは無音**です（会場スピーカーからのみ再生）
- 効果音はWeb Audio APIで合成されるため、外部ファイル不要
- ブラウザのオーディオポリシーにより、初回はユーザー操作（クリック等）後に再生が有効化されます

## 5. 参加者の操作手順

### Step 1: 参加する

1. QRコードをスマホのカメラで読み取る（または `/play` のURLをブラウザで開く）
2. 表示された画面でルームコード（6桁数字）を入力して「参加する」

### Step 2: プロフィール設定

1. ニックネームを入力（最大8文字、同じクイズ内で重複不可、制御文字は自動除去）
2. A〜Z のレター名チームから所属チームを選ぶ（必須）
3. カメラで自撮り撮影（必須）
4. 「参加する」ボタンをタップ

> **注意**: 同じニックネームが既に使われている場合はエラーメッセージが表示されます。別のニックネームを入力してください。
> チームを選択しないと「参加する」ボタンが押せません。

### Step 3: 待機〜ゲーム開始

- ホストがゲームを開始するまで待機画面が表示される
- ゲーム開始と同時に自動で回答画面に切り替わる

### Step 4: 回答する

- 問題文が表示されたら4色のボタンから答えを選ぶ
- 選択後はボタンが押せなくなる（変更不可）
- 制限時間内に回答しないと0点

### Step 5: 結果確認

- 正解・不正解と獲得ポイントが表示される
- ポイントは **早押しボーナス付き**（速く答えるほど高得点）
- 現在順位も確認できる

### 再接続について

- ブラウザを閉じてしまっても `localStorage` のトークンで自動復帰
- 同じURLに再アクセスすれば続きからゲームに参加できる

## 6. 各画面の詳細説明

### 6-1. ホスト画面一覧

#### `/host/setup` — ダッシュボードハブ

- クイズの新規作成（タイトル入力）
- 既存クイズの選択・編集・削除
- 問題の追加・編集・削除・ドラッグ&ドロップ並べ替え
- 問題バンク（テンプレート管理・クイズへのインポート）
- ゲーム履歴の閲覧（過去のクイズ結果・参加者一覧）
- 参加者ギャラリー（全参加者の自撮り一覧・個別/一括削除）
- 統計ダッシュボード（参加者数・正答率・平均スコア等のサマリー）
- データエクスポート（JSON / CSV 形式でクイズ結果をダウンロード）
- プレビューボタン（参加者視点で問題を確認）
- リハーサルボタン（テスト進行モード）
- ロビーを開くボタン

#### `/host/:roomCode?quizId=QUIZ_ID` — ホスト操作画面

フェーズごとの表示と操作：

| フェーズ | 表示内容 | 操作ボタン |
|----------|----------|-----------|
| lobby | QRコード・ルームコード・参加者一覧 | ゲーム開始 |
| countdown | 5秒カウントダウン | （自動進行） |
| question | 問題文・選択肢・カウントダウン・回答数 | 回答を締め切る |
| results | 回答分布グラフ・正解表示 | ランキングを表示 / 次の問題 |
| ranking | スコアバーアニメーション（10名ずつページ切替） | ページ切替 / 次の問題 / 最終結果発表（チーム戦時はモード切替も） |
| final | カウントダウン発表 → 集合写真（浮遊アバター） | もう一度プレイ / ゲーム終了 |
| closed | サンキュースクリーン（浮遊アバター） | 管理画面に戻る / ルームを削除（2段階確認） |
| recovering | ホスト復旧画面 | 次の問題を配信 / ランキング表示 |

#### `/host/:roomCode/screen` — プロジェクター専用表示画面

- ホスト操作画面と完全同期（読み取り専用）
- 操作ボタンは一切なし
- ゲーム開始後にアクセスしても既存状態を取得して同期

---

### 6-2. 参加者画面一覧

#### `/play` — ルームコード入力画面

- 6文字のルームコードを入力して参加

#### `/play/:roomCode` — 参加フロー画面

フェーズごとの表示：

| フェーズ | 表示内容 |
|----------|----------|
| profile | ニックネーム入力（8文字以内）+ チーム選択（A〜Z から選択）+ 自撮り撮影（必須） |
| waiting | 待機画面（ルームコード表示） |
| answer | 4色ボタンで回答・制限時間カウントダウン・回答数表示 |
| result | 正解/不正解・正解テキスト・獲得ポイント・現在順位（終盤5問は現在順位を非表示） |
| ranking | 自分の順位（大表示）+ 上位5名ミニランキング（終盤5問はこの画面をスキップ） |
| final | 自分の最終順位・正答数・平均回答速度 |
| closed | サンキュースクリーン（浮遊アバター） |

## 7. トラブルシューティング

### Q: 参加者がQRコードを読み取れない

**A:** 以下を確認してください
- プロジェクターへの投影が見づらい場合は、ホストPCの画面を直接見せるか、URLとルームコードを口頭で案内する
- QRコードに十分な余白があるか確認（プロジェクターの端が切れていないか）

---

### Q: 参加者の画面が「接続中…」から進まない

**A:**
1. 参加者のスマホが Wi-Fi に接続されているか確認
2. バックエンドサーバーが起動しているか確認（`http://<バックエンドURL>/api/quizzes` でレスポンスがあるか）
3. スマホのブラウザをリロードする

---

### Q: ホスト操作画面に「接続エラー」が表示される

**A:**
1. バックエンドサーバーが起動しているか確認
2. URLの `key=` パラメータが正しいか確認（SetupPage から遷移した場合は自動設定される）
3. ブラウザの localStorage をクリアして再度 SetupPage からやり直す

---

### Q: プロジェクター画面がホスト画面と同期しない

**A:**
1. プロジェクターPCのブラウザで `/host/:roomCode/screen` のURLが正しいか確認
2. ブラウザをリロードする
3. それでも解決しない場合は、ゲーム開始後でも `/screen` URLに再アクセスすれば現在の状態が取得される

---

### Q: 回答が送信できない（ボタンが反応しない）

**A:**
- ホストが問題を締め切った後は回答できません（正常な動作）
- 制限時間が切れた後も回答できません（正常な動作）
- 通信切れの場合はリロードして再接続してください

---

### Q: 参加者がブラウザを閉じてしまった

**A:**
- 同じURLに再アクセスすると `localStorage` のトークンで自動復帰します
- ゲーム途中でも再接続後、次の問題から参加できます

---

### Q: 画像がアップロードできない / 表示されない

**A:**
- ファイルサイズが5MB以上の場合はアップロードできません
- 対応フォーマット：JPG・JPEG・PNG・GIF・WebP のみ
- バックエンドの `uploads/` ディレクトリに書き込み権限があるか確認

---

### Q: 途中でサーバーが落ちてしまった

**A:**
1. バックエンドを再起動する
2. **ゲームの状態はメモリ上に保持されているため、再起動するとリセットされます**
3. ゲームを最初からやり直す必要があります
4. 本番では Azure App Service の「Always On」設定を有効にしておくことを推奨します

## 8. Azure デプロイ手順

### フロントエンド（Azure Static Web Apps）

1. Azure Portal で「Static Web App」を新規作成
2. GitHub リポジトリと連携（または手動デプロイ）
3. ビルド設定：
   - アプリの場所: `src/frontend`
   - ビルドコマンド: `npm run build`
   - 出力フォルダ: `dist`
4. 環境変数 `VITE_API_URL` にバックエンドのURLを設定

### バックエンド（Azure App Service B1 Linux）

1. Azure Portal で「App Service」を新規作成
   - ランタイム: Node.js 22 LTS
   - プラン: Basic B1（または S1）
   - OS: Linux
2. デプロイメント設定（GitHub Actions または ZIP デプロイ）
3. 環境変数を設定：
   - `NODE_ENV`: `production`
   - `PORT`: `8080`（App Service のデフォルト）
   - `CORS_ORIGIN`: フロントエンド URL の許可リスト
   - `ADMIN_PIN`: 管理画面アクセス用 PIN（production では設定必須）
4. スタートアップコマンド: `npm run db:migrate && npm start`
5. **CORS 設定**: フロントエンドの Static Web Apps URL を `CORS_ORIGIN` に設定

### SQLite データの扱い

- App Service の `/home` 以下は永続化されます（再起動後も保持）
- データディレクトリを `/home/data` に設定することを推奨
- 大規模な本番利用には Azure SQL や Turso（SQLite as a Service）への移行を検討

### 結婚式当日のみ使う場合

- App Service は **停止** にすると課金が止まります（コンピューティングのみ。ストレージは微課金）
- イベント前日に起動 → 翌日終了で実質 **2〜3日分** ≒ **¥200〜300** 程度

- ゲーム当日だけ App Service を起動する（平常時は停止）
- 参加者100人・1時間のゲームなら Free Tier の帯域制限（100GB/月）は超過しない
- 自撮り写真を多用する場合はストレージも確認（uploads/ ディレクトリ）

## 10. テスト手順

3段階のテスト方法があります。すべて**無料**のオープンソースツールで実行できます。

### 10-1. レベル1: 手動テスト

**今すぐできる。** 複数のブラウザタブを使って実際に操作確認します。

1. バックエンドとフロントエンドを起動（[4-1. サーバー起動](#4-1-サーバー起動) 参照）
2. タブを3〜4枚開く：

| タブ | URL | 役割 |
|------|-----|------|
| タブ1 | `http://localhost:5174/host/setup` | ホスト操作 |
| タブ2 | `http://localhost:5174/host/<ルームコード>/screen` | プロジェクター確認 |
| タブ3〜5 | `http://localhost:5174/play` | 参加者（スマホ模擬） |

**スマホ模擬の方法：** Chrome で `Ctrl+Shift+M`（Mac: `Cmd+Shift+M`）でデバイスエミュレーションを開き、iPhone/Android モードに切り替える。

**実機テストの方法：** PCとスマホを同じ Wi-Fi に接続して、スマホのブラウザで `http://<PCのLAN IP>:5174/play` にアクセスする（PCのIPは `ipconfig` で確認）。

---

### 10-2. レベル2: E2E テスト（Playwright）

ブラウザを自動操作してゲームフロー全体を検証します。

#### 前提条件

- バックエンドが起動していること（ポート 3001）
- フロントエンドが起動しているか、後述のコマンドが自動起動

#### テスト実行コマンド

```bash
cd src/frontend

# ヘッドレス実行（CIや高速確認向け）
npm run e2e

# ブラウザを表示しながら実行（動作を目で確認できる）
npm run e2e:headed

# Playwright UI モード（テストを1件ずつ選んで実行、デバッグに便利）
npm run e2e:ui

# HTML レポートを開く（直前の実行結果）
npm run e2e:report
```

#### テストファイル一覧

| ファイル | 内容 |
|----------|------|
| `e2e/join.spec.ts` | JoinPage の入力バリデーション（ルームコード6文字・大文字変換など） |
| `e2e/participant-flow.spec.ts` | ニックネーム入力・文字数制限 |
| `e2e/game-flow.spec.ts` | クイズ作成→ロビー→参加→ゲーム開始の完全フロー（`@integration`） |
| `e2e/404.spec.ts` | 存在しないURLで404ページが表示されること |
| `e2e/multiplayer.spec.ts` | ホスト＋参加者の同時接続フロー |

#### タグによる絞り込み実行

`@integration` タグのついたテストはバックエンドが必要です。タグで絞り込めます：

```bash
# バックエンド不要のテストだけ実行
npm run e2e -- --grep-invert "@integration"

# 統合テストだけ実行
npm run e2e -- --grep "@integration"
```

#### 結果の見方

- 緑 ✓：成功
- 赤 ✗：失敗（`playwright-report/` にスクリーンショット + トレースが保存される）
- `npm run e2e:report` でブラウザで詳細レポートを開ける

### 10-3. レベル3: 負荷テスト（Artillery）

100人が同時接続してもサーバーが落ちないか検証します。

#### 前提条件

- バックエンドが起動していること（ポート 3001）
- Node.js 18 以上

#### 初回セットアップ（1回だけ）

```bash
cd tests/load
npm install
```

#### テスト実行手順

**Step 1: テスト用ルームを作成する**

```bash
node setup.mjs
```

出力例：
```
=== Wedding Quiz 負荷テストセットアップ ===
[1/3] クイズを作成中...
[2/3] 問題を追加中...（3問）
[3/3] ルームを開設中...
  → ルームコード: ABCXYZ
✓ セットアップ完了！
```

内部でクイズ3問を自動作成し、Socket.io でロビーを開きます。ルームコードは `.env.load` に保存されます。

**Step 2: 負荷テストを実行する**

```bash
npm test          # 最大 ~50 同時ユーザー（ウォームアップあり）
npm run test:100  # 最大 ~100 同時ユーザー
```

実行中はリアルタイムで進捗が表示されます：

```
Summary report @ ...
```

**Step 3: レポートを確認する（任意）**

```bash
npm run report
# → report.html が生成されてブラウザで開ける（グラフ付き詳細レポート）
```

#### 見るべき指標

| 指標 | 合格基準 | 意味 |
|------|---------|------|
| `vusers.failed` | 0 | 接続エラーなし |
| `http.response_time.p95` | 500ms 以内 | 95% のリクエストが 500ms 以内 |
| `http.response_time.p99` | 1000ms 以内 | 99% のリクエストが 1秒以内 |
| `socketio.emit.rate` | 全ユーザー分 | joinRoom が全員分送信されている |

#### テスト後のクリーンアップ

```bash
node setup.mjs --cleanup
# → テスト用クイズをDBから削除
```
