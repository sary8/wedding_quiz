# Wedding Quiz App 機能仕様書

> バージョン: 1.3.0
> 最終更新: 2026-03-04
> ステータス: 確定

---

## 概要

結婚式の二次会用 Kahoot! 形式リアルタイムクイズアプリ。
ホスト（PC/プロジェクター）から問題を出し、参加者約100人がスマホから回答する。

---

## F001: クイズ管理

### 概要
ホストがクイズの作成・編集・削除・問題管理を行う。

### 受入基準
- [ ] クイズの新規作成（タイトル入力）
- [ ] クイズタイトルの編集（インライン編集）
- [ ] クイズの削除（cascade で問題・参加者・回答も削除）
- [ ] 問題の追加（問題文・選択肢4択・正解番号・制限時間5-120秒・画像任意）
- [ ] 問題の編集（全フィールド更新可能）
- [ ] 問題の削除
- [ ] 問題の並べ替え（D&D + ↑↓ボタン）
- [ ] 問題文: 最大500文字、選択肢: 最大200文字
- [ ] 画像: JPG/PNG/GIF/WebP、最大5MB、magic byte検証
- [ ] mediaType: "image" のみ（動画UIは実装しない）

### UI
- SetupPage ダッシュボードハブ（作成・一覧・履歴・ギャラリー・ライブラリ）
- QuestionManagementTab（@dnd-kit D&D + 手動↑↓ボタン）

## F004: ゲーム進行

### 概要
ホストが問題を配信し、参加者がリアルタイムで回答する。

### ゲームフロー
```
lobby → countdown(5秒) → question → results → ranking → [次の問題 or 最終発表]
                                                              ↓
                                                     final → group → closed
```

### 受入基準
- [ ] 5秒カウントダウン後に最初の問題を自動配信
- [ ] サーバーサイドタイマー（hrtime高精度、クライアント改ざん不可）
- [ ] 4色回答ボタン（赤・青・緑・黄）
- [ ] 回答二重送信防止（UNIQUE制約 question_id + participant_id）
- [ ] 手動締め切り or タイマー自動締め切り
- [ ] 回答分布バーチャート（結果フェーズ）
- [ ] 参加者に正解テキスト表示
- [ ] ライブ回答数表示（参加者画面にも）
- [ ] タイマー値クランプ（Math.max(0, remaining)）

### スコアリング
- 正解: 500〜1000点（速度ボーナス、早いほど高得点）
- 不正解: 0点
- 計算式: `baseScore * (1 - responseTime / timeLimit * 0.5)`

### Socket.io
- `startGame` → `gameStarted`
- `nextQuestion` → `questionStarted`
- `timeUpdate` — 毎秒残り時間
- `answerCountUpdate` — リアルタイム回答数
- `submitAnswer` — 回答送信
- `closeQuestion` → `questionClosed` → `questionResult`

- [ ] 集合写真フェーズ: 浮遊アバターアニメーション（スクリーンセーバー風）
- [ ] prefers-reduced-motion時は静的グリッド表示

### Socket.io
- `endGame` → `gameEnded`

## F017: 画像選択肢

### 概要
問題の選択肢にテキストだけでなく画像を使えるようにする。

### 受入基準
- [ ] 問題ごとに「テキスト選択肢」か「画像選択肢」を切り替え（混在なし）
- [ ] 画像選択肢にはオプションでテキストラベルを付けられる（最大200文字）
- [ ] 既存の問題レベル画像（`media_url`）と画像選択肢は併用可能
- [ ] 既存の問題はすべて `choice_type = "text"` として後方互換
- [ ] テンプレート（問題バンク）でも画像選択肢をサポート
- [ ] テンプレートからインポート時に画像URLも保持

### DB変更
`questions` と `question_bank` テーブルに以下のカラムを追加:
- `choice_type` TEXT NOT NULL DEFAULT 'text' — "text" | "image"
- `choice1_image_url` TEXT — nullable
- `choice2_image_url` TEXT — nullable
- `choice3_image_url` TEXT — nullable
- `choice4_image_url` TEXT — nullable

### バリデーション
- `choice_type === "text"`: 従来通り choice1-4 テキスト必須
- `choice_type === "image"`: choice1-4 テキストは任意（ラベル）、代わりに image_url 4つ必須
- 画像フォーマット: JPG/PNG/GIF/WebP、最大5MB（問題画像と同一制約）
- magic byte検証あり

### 画像ガイドライン
- 推奨アスペクト比: 正方形（1:1）または 4:3
- 推奨解像度: 最低500×500px
- ホスト画面では `object-cover` でクロップ表示されるため、被写体を中央に配置推奨
- 参加者画面では `object-cover` でボタン内に収まる表示

### UI

#### 問題作成（QuestionInlineForm）
- テキスト/画像の切替トグルボタン
- 画像モード時: 各選択肢に画像アップロード UI + オプションのテキストラベル入力
- アップロード中は「アップロード中…」表示、完了でサムネイル表示
- 切替時のデータ保持: テキスト↔画像を切り替えても入力済みデータは保持される（元に戻せば復元可能）
- 保存条件: 画像モードでは4つの画像すべてアップロード完了が必須

#### ホスト/プロジェクター画面（QuestionPage）
- 2×2グリッドに画像サムネイル（112-144px）+ テキストラベル
- 画像ありの場合フォントサイズを縮小して画像スペースを確保
- `object-cover` + 角丸で統一表示

#### 参加者画面（AnswerPage / ChoiceButton）
- 2×2グリッドで画像 + オプションラベル
- 画像はボタン内で `object-cover` 表示
- 最小タッチターゲット 44px 維持

#### 結果画面（ResultsPage）
- 分布バーチャートの左に画像サムネイル（56-80px）を表示
- `sanitizeMediaUrl()` でURL検証済みの画像のみ表示

#### プレビュー画面（PreviewPage）
- 参加者画面と同じレイアウトで画像選択肢をレンダリング
- 正解の選択肢をハイライト表示

### エラーハンドリング
- 画像読み込み失敗時: テキストラベルにフォールバック（ラベル未設定なら「選択肢A」等）
- アップロード失敗時: エラートースト表示（「選択肢Xの画像アップロードに失敗しました」）
- `sanitizeMediaUrl()` が null を返した場合: 画像タグを非表示にしてテキストのみ表示

## F018: ○×問題（2択問題）

> ステータス: 実装済み | 優先度: P1

### 概要
通常の4択に加えて、○（正しい）か×（間違い）の2択で回答する問題形式を追加する。

### 受入基準
- [x] 問題作成時に問題形式を選択: 「4択」「○×」
- [x] ○×問題は正解が「○」か「×」の2択
- [x] 参加者画面: 画面左半分に大きな○ボタン、右半分に大きな×ボタン
- [x] ホスト/プロジェクター画面: ○と×を大きく2つ表示
- [x] 結果画面: ○と×の回答分布を表示
- [x] スコアリングは4択と同じ計算式（速度ボーナスあり）
- [x] 問題バンク（テンプレート）でも○×問題をサポート
- [x] プレビュー画面で○×問題の表示に対応

### DB変更
`questions` と `question_bank` テーブル:
- `question_type` TEXT NOT NULL DEFAULT 'four_choice' — "four_choice" | "true_false"
- ○×問題時: `choice1` = "○", `choice2` = "×", `choice3`/`choice4` = null, `correct_choice` = 1 or 2

### UI

#### 参加者画面（AnswerPage）
- 2ボタンレイアウト（左右分割、`grid-cols-2`）
- ○ボタン: パステルグリーン系、大きな○マーク
- ×ボタン: パステルローズ系、大きな×マーク
- 最小タッチターゲット: 44px以上（実質画面半分なので十分）

#### ホスト/プロジェクター画面
- 2ボタンレイアウトで○と×を大きく表示
- 回答分布も○/×の2本バー

### 影響範囲
- DB: schema.ts (question_type カラム追加)
- Backend: routes/question.ts, routes/questionBank.ts (バリデーション分岐)
- Frontend: QuestionInlineForm, ChoiceButton, AnswerPage, QuestionPage, ResultsPage, PreviewPage

## F019: ポイント倍率問題

> ステータス: 実装済み | 優先度: P1

### 概要
特定の問題のポイントを倍にする「ボーナス問題」機能。事前設定とゲーム中指定の両方に対応。

### 受入基準
- [x] 問題作成時に倍率を設定可能: 1x（デフォルト）, 2x, 3x
- [ ] ゲーム中にホストが出題時に倍率を変更可能
- [x] 倍率適用時、参加者画面に「ボーナス問題！」「ポイント2倍！」等のバナー表示
- [x] ホスト/プロジェクター画面にも倍率バッジ表示
- [x] スコア計算: 通常スコア × 倍率
- [x] ランキングに倍率分のスコアが正しく反映
- [x] ゲーム履歴・統計にも倍率情報を保持

### DB変更
`questions` テーブル:
- `point_multiplier` INTEGER NOT NULL DEFAULT 1 — 1, 2, or 3

### スコアリング
```
finalScore = baseScore * (1 - responseTime / timeLimit * 0.5) * pointMultiplier
```
- 正解: (500〜1000) × 倍率
- 不正解: 0点（倍率に関係なく）

### UI

#### 問題作成（QuestionInlineForm）
- 倍率セレクター: 1x / 2x / 3x（ラジオボタンまたはセグメント）
- 2x以上の場合は「ボーナス問題」バッジ表示

#### ゲーム中のホスト操作
- 出題前に倍率変更ボタン（ドロップダウンまたはトグル）
- 変更はその問題にのみ適用

#### 参加者画面
- 倍率2x以上の問題: 画面上部に「ボーナス問題！ポイントX倍！」バナー
- バナーの色: ゴールド系

#### ホスト/プロジェクター画面
- 問題表示時に倍率バッジ（「×2」「×3」）
- 結果画面でも倍率込みのスコアを表示

### Socket.io
- `questionStarted` ペイロードに `pointMultiplier` フィールドを追加
- `setMultiplier` — ホストがゲーム中に倍率を変更するイベント（任意）

## F020: チーム対戦モード

> ステータス: 実装済み | 優先度: P0

### 概要
参加者をチームに分けて、チーム単位でも競い合えるモード。個人ランキングとチームランキングの両方を表示する。

### 受入基準
- [x] ホストがクイズ設定でチームモードを有効化できる（トグルスイッチ）
- [x] ホストがチーム数とチーム名を事前設定（2〜10チーム）
- [x] 参加者が参加時にチームを選択（ボタン形式）
- [x] チーム選択UIをプロフィール画面に追加
- [x] ゲーム中の回答・スコアリングは個人単位（通常と同じ）
- [x] ランキング画面: チームランキング（上部）+ 個人Top10（下部）の両方を表示
- [x] チームスコア = メンバーの合計スコア
- [x] 最終結果発表: teamReveal（チームランキング）→ scroll → top3 → done → group
- [x] 参加者スマホに自分のチーム順位も表示
- [x] チーム非使用時は従来通りの個人戦のみ（後方互換）
- [x] リプレイ時もチーム割り当て維持（スコアのみリセット）

### DB変更

`quizzes` テーブル:
- `team_mode` INTEGER NOT NULL DEFAULT 0 — チームモードON/OFF

新テーブル `teams`:
- `id` INTEGER PRIMARY KEY
- `quiz_id` INTEGER NOT NULL (FK → quizzes, ON DELETE CASCADE)
- `name` TEXT NOT NULL
- `order_index` INTEGER NOT NULL DEFAULT 0

`participants` テーブル:
- `team_id` INTEGER — nullable (FK → teams, ON DELETE SET NULL)

### チーム管理

#### ホスト設定（QuizConfigTab）
- クイズ設定カード内に「チーム戦モード」トグル（role="switch"）
- ON時: チーム名入力フォーム（最低2、最大10チーム）
- デフォルトチーム名: 「チーム1」「チーム2」（編集可能）
- 「+ チーム追加」ボタン / 「×」で削除
- 「チームを保存」ボタン → PUT `/api/quizzes/:id/teams`

#### 参加者のチーム選択（ProfilePage）
- チームモード有効時: ニックネーム入力の下にチーム選択ボタン一覧
- 選択中のチームは ring で強調表示
- チーム未選択では「参加する」ボタン disabled

### ランキング表示

#### ホスト/プロジェクター画面（RankingPage）

## F023: フォトモザイク

> ステータス: 未実装 | 優先度: P2

### 概要
参加者全員の自撮りアバターを使って、ベース画像（新郎新婦の写真やテキスト/ロゴ）をモザイクアートとして再構成する記念機能。ゲーム終了後に自動生成し、プロジェクターに表示。ホストがダウンロード可能。

### 受入基準
- [ ] ホストがクイズ設定でモザイク用ベース画像をアップロードできる（任意）
- [ ] ベース画像の代わりにテキスト/ロゴ入力も可能（Canvas描画で画像化）
- [ ] ベース未設定時はモザイク機能をスキップ
- [ ] ゲーム終了（`endGame`）後、サーバー側でモザイク画像を自動生成
- [ ] 生成完了後、プロジェクター/ホスト画面に表示（集合写真フェーズの後）
- [ ] ホストが「ダウンロード」ボタンで画像を保存可能
- [ ] 参加者が5人未満の場合はモザイク生成をスキップ（タイルが少なすぎる）

### モザイク生成仕様

#### 入力
- ベース画像: ホストアップロード（JPG/PNG、最大10MB）またはテキスト→Canvas→PNG変換
- タイル画像: 全参加者の自撮りアバター（`avatar_url`）

#### 処理（バックエンド: Sharp）
1. ベース画像を1920×1080にリサイズ（アスペクト比維持、`cover`）
2. ベース画像をグリッドに分割（タイル数 = 参加者数に応じて最適計算）
3. 各グリッドセルの平均色を算出
4. 各タイル画像を正方形にクロップ・リサイズ
5. 各セルに最も色が近いタイル画像を配置（同じ画像の連続使用を避ける）
6. タイル画像の上にベース画像をアルファブレンド（透明度30-50%でモザイク感を出す）
7. 最終画像を1920×1080 JPEGで出力（品質85%）

#### タイルサイズ計算
- 参加者数に応じてタイルサイズを自動調整
- 目安: 20人→96px, 50人→60px, 100人→48px
- タイル数が参加者数より多い場合、タイルを繰り返し使用

#### 出力
- 保存先: `uploads/mosaic_{quizId}_{timestamp}.jpg`
- 解像度: 1920×1080（プロジェクター最適化）
- 生成時間目安: 100人で3-5秒（B1インスタンス）

### DB変更
`quizzes` テーブル:
- `mosaic_base_url` TEXT — nullable（ベース画像URL）
- `mosaic_base_type` TEXT — nullable（"image" | "text"）
- `mosaic_base_text` TEXT — nullable（テキストモード時の文字列）
- `mosaic_result_url` TEXT — nullable（生成結果URL）

### API
- `PUT /api/quizzes/:id/mosaic-base` — ベース画像/テキスト設定
- `GET /api/quizzes/:id/mosaic` — モザイク画像取得
- `DELETE /api/quizzes/:id/mosaic-base` — ベース画像削除

### Socket.io
- `mosaicReady` — モザイク生成完了通知（URLを含む）

### UI

#### ホスト設定（QuizConfigTab）
- 「フォトモザイク」セクション（チーム設定の下）
- トグル: 画像アップロード / テキスト入力 の切替
- 画像モード: ファイル選択 + プレビュー
- テキストモード: テキスト入力（最大20文字）+ フォント色 + 背景色
- 「プレビュー」ボタンでテキスト→画像変換結果を確認

#### プロジェクター/ホスト画面（MosaicRevealPage）
- 集合写真フェーズの後に自動遷移
- 生成中: ローディングアニメーション + 「モザイクアート生成中...」
- 完成: ズームインアニメーション（scale 0.5→1, duration 2s）
- ホスト画面: 「ダウンロード」ボタン表示

#### ゲームフロー変更
```
... → final → group → mosaic（新規） → closed
```
- mosaic フェーズはベース画像設定済みかつ参加者5人以上の場合のみ
- 未設定の場合は group → closed（従来通り）

### 依存関係
- `sharp` パッケージを backend に追加
- Sharp のネイティブバイナリがAzure App Service B1 Linux で動作すること

### エラーハンドリング
- Sharp インストール失敗時: モザイク機能を無効化（graceful degradation）
- ベース画像の読み込み失敗: エラーログ出力、モザイクフェーズをスキップ
- 生成タイムアウト（30秒）: スキップしてclosedフェーズに進む

### エッジケース定義

#### EC-1: 回答中の接続断
| 状況 | 期待動作 |
|------|---------|
| 回答ボタン押下 → サーバー到達前に切断 | 回答は記録されない。再接続後、問題がまだ開いていれば回答可能。閉じていれば無回答扱い（0点） |
| 回答ボタン押下 → サーバー到達後に切断 | 回答は記録済み。再接続後、`reconnected` イベントで現在の状態（回答済み/結果待ち）を復元 |
| タイマー残り中に切断 → タイムアウト後に再接続 | 問題は締め切り済み。再接続時に `questionResult` を受信し結果画面を表示 |
| ロビー中に切断 → ゲーム開始後に再接続 | `reconnected` イベントで現在の問題フェーズに復帰。切断中の問題は無回答扱い |

**実装要件:**
- 参加者のSocket切断から再接続まで `quiz_token_{roomCode}` (localStorage) で同一参加者を識別
- 再接続時、サーバーはその参加者の最新状態（フェーズ、回答済み問題、スコア）を送信
- 切断中にゲームが `closed` になった場合、再接続時に `gameClosed` を送信

#### EC-2: 100人同時回答
| 状況 | 期待動作 |
|------|---------|
| 100人が1秒以内に一斉回答 | 全回答が正しく記録される。UNIQUE制約で二重送信を防止 |
| 回答の順序保証 | `response_time_ms` はサーバー受信時刻 - 問題開始時刻で計算。ネットワーク遅延は考慮しない（会場LAN前提で均一） |
| answerCountUpdate の正確性 | 各回答処理後にカウントをインクリメント。バッチではなくリアルタイムに通知 |
| サーバー負荷 | 100件/秒の書き込みでSQLiteがボトルネックにならないこと（WALモード使用） |

**実装要件:**
- SQLite WALモード有効化（同時読み書き対応）
- `submitAnswer` ハンドラは非同期・並列処理可能
- 回答処理失敗時はクライアントに `answerError` を返し、リトライを許可しない

#### EC-3: ブラウザバック / タブ切替
| 状況 | 期待動作 |
|------|---------|
| 参加者が回答画面でブラウザバック | ルーム参加画面に戻る。Socket接続は維持。次にフォワードで戻ると現在のフェーズを表示 |
| 参加者がタブを切り替えて戻る | Socket接続は維持（ブラウザがバックグラウンドでもWebSocketは切断しない）。画面復帰時にUIは最新状態を表示 |
| 参加者がタブを閉じる | Socket切断。ただしサーバーは5分間参加者データを保持（ゴースト削除タイマー）。同じトークンで再アクセスすれば復帰可能 |
| ホストがブラウザバック | SetupPage に戻る。Socket切断。再度ルームを開く場合は `openRoom` で再接続。ゲーム中の状態はサーバーに保持されている |
| iOS Safari でホーム画面に戻る | バックグラウンド30秒後にSocket切断。再度開くと自動再接続を試行 |

**実装要件:**
- `visibilitychange` イベントでページ復帰時にSocket接続状態を確認
- 切断検知時は自動再接続（Socket.ioのデフォルト再接続 + カスタムリカバリ）
- ブラウザバックで状態が失われないよう、`sessionStorage` にフェーズ情報を保持

#### EC-4: ホストPCスリープ / 画面ロック
| 状況 | 期待動作 |
|------|---------|
| ホストPCがスリープ | Socket切断。サーバーのタイマーは継続動作。復帰後に `openRoom` で再接続し、現在の状態を復元 |
| 問題出題中にスリープ → 復帰 | `hostReconnected` イベントで currentQuestionData, answerCount, timerRemaining を含む状態を復元。タイマーが0以下なら自動締め切り |
| プロジェクターPCがスリープ | `watchRoom` で再接続。現在の全状態を取得 |
| ホストPCの画面ロック（ロック画面表示） | WebSocket接続は通常維持される。画面ロック解除後にUIは最新状態を表示 |

**実装要件:**
- ホスト再接続時: サーバーは `hostReconnected` ペイロードに以下を含める
  - `phase`: 現在のフェーズ
  - `currentQuestionData`: 出題中の問題データ
  - `answerCount`: 現在の回答数
  - `timerRemaining`: 残り秒数（hrtime計算）
  - `rankings`: 最新ランキング（結果/ランキングフェーズ時）
- `Page Visibility API` を使って復帰を検知し、必要に応じて再接続
- サーバー側ゲーム状態はメモリに保持（Socket切断で破棄しない）
